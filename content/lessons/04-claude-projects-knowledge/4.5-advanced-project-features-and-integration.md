## Lesson script

### When one Project is no longer enough

You set up your first Project. You added your business context, a system prompt, and a few key documents. You ran a task and the output was noticeably better than what you were getting before — because Claude already knew your business when the conversation started.

That's the foundation. This lesson covers what happens next: when your Project library grows past one, when other people in your business start needing access, when conversations accumulate and you have to manage them, and when you want Claude's output to end up somewhere useful rather than just sitting in a chat window.

None of this requires technical skills. It does require a few clear decisions. This lesson gives you the framework for those decisions.

### Sharing Projects with team members

Claude Projects on a Team or Pro plan can be shared with other users in your account. Shared Projects mean one system prompt, one set of uploaded documents, and one consistent Claude behaviour — for every person in the Project.

That is the core value of a shared Project. Not convenience. Not fewer logins. It's the fact that when your bookkeeper opens the Accounts Project, they're working with the same context as when you open it. Claude knows your chart of accounts, your payment terms, your supplier names, the way you phrase things to clients. You don't have to brief Claude separately for each person. The Project does the briefing for everyone who has access.

When does sharing make sense? When two or more people in your business regularly do the same type of Claude task. A shared Client Communications Project makes sense if you have two people handling client emails and you want them both sounding like the same business. A shared Onboarding Project makes sense if you have multiple people who train new staff. A shared Content Project makes sense if more than one person writes copy or social posts.

When does sharing not make sense? When a Project contains information only one person should see — financial details, HR matters, client contracts with confidentiality clauses. Keep those as personal Projects. One person, one set of private documents, no shared access.

Permissions on shared Projects are set by the Project owner. In most Claude plans, you can add or remove team members, and all members can start conversations and read Project knowledge. Only the owner can change the system prompt or remove documents. Check your plan's specific permissions before sharing — they vary.

The practical benefit of a shared Project is what you might call single source of truth. When you update the system prompt — because your services changed, or your pricing updated, or your brand tone shifted — it updates for everyone in the Project immediately. No need to send an updated brief to each person. No risk that one person is running on old instructions while another has the new ones. One edit, consistent behaviour.

### Using multiple Projects together

By the time you finish Module 4 you might have two, three, or four Projects. A Client Communications Project. A Proposals Project. An Internal Operations Project. Maybe one for a specific seasonal campaign.

The critical skill is not how to set Projects up. It's how to decide which Project to open before you start a task.

The decision is faster than it sounds. Ask yourself one question: what standing knowledge does Claude need for this task to be useful?

If you're drafting an email to a client, you need Claude to know your communication tone, your service descriptions, and ideally the context of that particular client relationship. That points to a Client Communications Project.

If you're writing a job estimate, you need Claude to know your rate card, your service inclusions, and your standard terms. That points to a Proposals or Quotes Project.

If you're writing a staff update, you need Claude to know your team structure, your internal voice, and the policies you've already documented. That points to an Internal Operations Project.

The question is not "which Project has the most information?" It's "which Project has the right information for this specific task?" A Project with too much irrelevant information can create noise. A Project focused on the task at hand lets Claude apply its context cleanly.

If a task genuinely doesn't fit any of your Projects — it's a one-off, it doesn't need standing knowledge, it's something you'll never repeat — run it in a standard Claude conversation, not in a Project. Projects are for recurring task types. One-off tasks don't benefit from a Project's context overhead.

```
Before opening Claude for any task, I choose my Project with this habit:
Task type → Client comms → open Client Communications Project
Task type → Quote or proposal → open Proposals Project
Task type → HR or team → open Internal Operations Project
Task type → One-off → open standard Claude conversation
```

That decision tree costs ten seconds. It saves the confusion of dumping a task into the wrong Project and getting output that sounds like it's for the wrong audience.

### Managing conversation history inside a Project

Every conversation you start inside a Project adds to the conversation list in that Project. Over time, that list grows. This is normal. The question is when to start a new conversation versus when to continue an existing one.

The rule is simpler than you might expect.

Start a new conversation when the task is new — when you're working on something different, even if it's in the same general area. Working on a new client email starts a new conversation. Working on a different quote starts a new conversation. Starting a new week's batch of social posts starts a new conversation.

Continue an existing conversation when you're working on the same specific thing — when your next prompt directly references output from the previous one. Revising a draft you just generated, continuing a task partway through, or following up on a specific question Claude answered — these are all reasons to continue.

The reason this matters is context. Within a single conversation, Claude remembers everything said — the documents you mentioned, the decisions you made, the revisions you asked for. Across conversations, it doesn't. The Project's system prompt and uploaded documents carry across all conversations. Specific conversational context doesn't.

This means if you started a conversation last week to draft a client proposal and left it at 60% complete, opening a new conversation and saying "continue the proposal" won't work — Claude won't have access to what was generated in the old conversation. You'd need to open the old conversation and continue from there.

A practical naming habit helps here. Give each conversation a clear title as soon as you start it. Most Claude interfaces let you rename conversations. "Baker Street Café — quote August" is easier to find and continue than "New conversation 14." Ten seconds of naming saves five minutes of scrolling.

When should you archive or delete old conversations? When the task is fully done. When the output has been sent, saved, or delivered, you don't need the conversation anymore. An accumulation of finished conversations adds scroll time without adding value. Clear out the done ones regularly — once a month is usually enough.

### Common mistake → Better approach

**Mistake:** Using one Project for every type of task, because it's simpler than managing several.

A business owner creates one Project called "Business" and uploads every document they have — brand guide, service descriptions, price list, team roster, supplier contacts, client correspondence templates. The system prompt tries to cover everything: tone, industry, clients, format preferences for every output type. The Project becomes a general-purpose context that is partially useful for every task but precisely right for none. Claude is working from a system prompt that covers too much ground, and the document library is too broad to be applied cleanly to any single task.

**Better approach:** One Project per recurring task type. A Client Communications Project with just the documents relevant to client work. A Proposals Project with rate card, service descriptions, and standard terms. An Internal Project with team-facing documents. Each system prompt is short and focused. When you open a Project, the context is immediately relevant to the task at hand. You might have three or four Projects rather than one, but each one works better than a single overloaded Project ever could.

### Project limits and how to manage them

Claude Projects have practical limits. Document count and total file size are capped — the specific numbers depend on your plan. Conversations accumulate over time. And within a single conversation, context has a length limit: if a conversation runs long enough, the earliest messages fall out of Claude's active memory.

Here's how to manage each one.

**Document limits.** When you're approaching the document cap, resist the urge to merge files into one giant document. That creates its own problems — a single enormous file is harder for Claude to apply selectively. Instead, audit what's already in the Project. Remove documents that are outdated or that you never actually reference in tasks. Keep the library lean and current rather than comprehensive and stale. You almost always need fewer documents than you think, not more.

**Long conversations.** When a conversation runs across many prompts and rounds of revision, the earliest context starts to fade. If you notice Claude forgetting something it knew earlier in the conversation — an instruction you gave, a decision you made — that's the signal. At that point, the most effective fix is to start a new conversation and open it with a brief summary of where you are:

```
I'm continuing work on the Northside Plumbing service agreement.
We've agreed on: 12-month term, monthly billing, 30-day termination notice.
Still to complete: the scope of work section and the payment terms clause.
Please draft the scope of work section next, using our standard service
descriptions from the Project knowledge.
```

That summary re-establishes the key decisions without requiring Claude to remember the whole previous conversation. You keep the momentum. You don't start from scratch.

**Context management as a habit.** Treat each conversation like a work session. When you close a task, paste the final output somewhere permanent — your CRM, your document folder, your email drafts. Don't rely on Claude's conversation history as your storage system. The conversation is a working space, not a filing cabinet.

### Integrating Projects with external tools — no code required

Claude doesn't connect directly to your CRM, your project management tool, or your email system without additional technical setup. But that doesn't mean Claude stays isolated. The most practical integration approach for small businesses is structured copy-paste — and it works well.

The pattern is this: bring information from your external tool into Claude, get Claude's output, then take that output back into your external tool. No code. No plugins. No IT involvement.

Here are the most common versions of this workflow.

**From email to Claude and back.** Copy the text of a client email, paste it into your Project conversation with a clear instruction ("Draft a reply to this email in our standard client tone, addressing each of the three questions"), read Claude's draft, copy it, paste it into your email reply. The Project handles the standing context — your tone, your services, your terms. Your prompt handles the specific task. Your email client handles the send.

**From a document to Claude and back.** Copy the relevant section of a contract, a brief, or a client form into Claude with a specific task ("Summarise the key obligations from this contract section in plain English — what do we have to do, and by when"). Take the summary, paste it into your project management tool as a task note. Claude processed the document. Your tool stores the output. You did no technical integration.

**From your CRM to Claude and back.** Copy a client record or a set of meeting notes from your CRM, paste into Claude ("Based on these meeting notes, draft a follow-up email and a list of three next actions"). Take Claude's follow-up email and paste it into your email. Take the three next actions and paste them into your CRM or task manager. The Project's client communications context does the heavy lifting. Your copy-paste moves the information between systems.

This approach has limits. It's manual. It doesn't scale to hundreds of tasks per day. But for a small business running twenty to fifty Claude tasks per week across a team, structured copy-paste is efficient, reliable, and requires no technical setup. It's also easier to review and correct than automated integrations that run without human oversight.

When does automation make sense? When you're running the same workflow multiple times every day, the inputs are always in the same format, and the output goes to the same place every time. At that point, it's worth talking to someone about a lightweight integration — tools like Zapier or Make can connect Claude to your CRM or email without code. But that's a future step, not a Module 4 step.

For now, structured copy-paste is the right tool. It puts a human in the loop, which is appropriate for client-facing output. It requires no setup. And it works well with Projects precisely because the Project carries the standing context — you're not re-briefing Claude every time you paste something in. The Project already knows your business. You're just bringing it the raw material for the current task.

## Worked examples

### Example 1: Sharing a Project with a second team member

**Scenario:** A café owner has set up a Client Communications Project and wants to give their floor manager access so they can handle supplier emails in the same tone.

**Prompt:**
```
We've added our floor manager Sam to this Project. Sam handles all supplier
communications — ordering, delivery scheduling, invoice queries. I handle
client-facing communications.

Update the system prompt to cover both use cases: when Sam asks about
supplier communications, Claude should know our supplier terms (net 30,
preferred delivery days are Tuesday and Thursday, we dispute invoices over
$500 before paying). When I ask about client communications, Claude should
keep using the existing tone and service context.

Draft the updated system prompt now.
```

**Why it works:** Rather than creating a second Project, the owner has expanded the existing one to cover a second task type for a second person. The prompt gives Claude a clear way to distinguish the two use cases, so the system prompt can address both without the guidance for one contaminating the guidance for the other. The standing context is now shared rather than duplicated.

### Example 2: Choosing the right Project for an edge-case task

**Scenario:** A bookkeeper with three Projects — Accounts, Client Comms, and Compliance — needs to write a letter to a client about an overdue invoice. She's not sure which Project to use.

**Prompt:**
```
I need to write a letter to a client — it's about an overdue invoice, so
it's partly financial and partly client relationship. Should I open Accounts
or Client Comms for this?

Think through which Project's standing context is more relevant: the Accounts
Project knows our payment terms and how we describe outstanding amounts.
The Client Comms Project knows our client communication tone and how we
handle sensitive situations. Which one matters more for a letter like this?
```

**Why it works:** The student uses Claude to reason through the Project selection — not to pick randomly, and not to open both. The framing ("which one matters more for a letter like this?") asks Claude to apply its knowledge of the two Projects' content to the specific task. The answer will almost always be Client Comms, because tone and relationship management dominate a collections letter — but thinking through it explicitly is a habit worth building.

### Example 3: Starting a new conversation vs. continuing an old one

**Scenario:** A property manager has an existing conversation about a lease renewal for one tenant, and now needs to draft a renewal for a different tenant.

**What to do — start a new conversation:**
```
Starting a new conversation in the Lease Documents Project.

I need a lease renewal letter for our tenant at 22 Fairview Street, Mooroolbark.
Current lease expires 30 June 2026. We're offering a 12-month renewal at
$1,850/month (up from $1,750). Same terms otherwise. Tenant has been with us
four years — good relationship. Friendly but professional tone.
```

**Why it works:** The previous conversation covered a different property and a different tenant. Continuing it would mix unrelated lease content in a way that could bleed into Claude's responses. Starting a new conversation keeps the work clean and the conversation history searchable by property. The Project context (standard lease language, property management tone) carries across without needing to be re-stated.

### Example 4: Recovering a long conversation with a summary opener

**Scenario:** A consultant's proposal conversation has run to 30+ prompts. Claude has started to forget earlier decisions. The consultant starts a new conversation with a recovery prompt.

**Prompt:**
```
I'm continuing work on the Meridian Group proposal. New conversation —
here's where we are:

Agreed: 3-month engagement, $18,500 fixed fee, fortnightly check-ins,
deliverables are a process audit report and an implementation roadmap.
Client's main concern: they don't want a generic audit — they want
recommendations specific to their industry (aged care).

Completed sections: executive summary, engagement scope, team bios.
Still to draft: methodology, timeline, investment summary.

Please draft the methodology section next. Use the standard language from
our Proposals Project for process audits, and frame it around aged care
operations specifically.
```

**Why it works:** The summary gives Claude the essential state of the project — what's decided, what the client's concern is, what's done, what remains — without requiring it to access the old conversation. The reference to "standard language from our Proposals Project" tells Claude to draw on the standing Project knowledge rather than inventing methodology language from scratch.

### Example 5: Structured copy-paste integration with a CRM

**Scenario:** A financial adviser copies meeting notes from her CRM into her Client Comms Project, gets Claude's output, then pastes the results back.

**Prompt:**
```
These are my notes from a discovery meeting this morning — copied from my
CRM:

Client: Thomas and Nadia W., both 58.
Meeting focus: retirement planning.
Key details discussed: Thomas retires in 4 years, Nadia in 6. Current super:
Thomas $420k, Nadia $310k. Own home outright. No debt. Want $7,500/month in
retirement. Concerned about longevity — Nadia's mother is 94.

Please do two things:
1. Draft a follow-up email to Thomas and Nadia that confirms what we discussed
and outlines the three areas we'll address in our next meeting (super strategy,
Centrelink eligibility timing, and estate planning considerations). Use our
standard client tone.
2. Write three CRM action items in a simple list — what I need to do before
the next meeting.

I'll copy the email into my email client and paste the action items back
into their CRM record.
```

**Why it works:** The prompt describes the copy-paste workflow explicitly — not to teach Claude the workflow, but to clarify that two distinct outputs are needed for two distinct destinations. The CRM notes arrive as plain text. The output leaves as a drafted email and a task list. The Project provides the standing context (client tone, practice services, communication standards) so the prompt only has to carry the meeting-specific facts.

## Graded deliverable

**Title:** Multi-step task through a Project — full conversation

**Brief:** Set up a second Project or substantially extend your existing one with at least one new document or a revised system prompt. Then run a real, two-turn task through it — not a demonstration or a test, but something you actually need done for your business. The task must require the Project's standing context to work well: if Claude could do it equally well in a standard conversation without any Project knowledge, you've chosen the wrong task.

Submit the two-turn conversation in full, plus a short reflection. Everything goes into the **Submission box** at the bottom of this lesson on the Claude Mastery platform. The AI grading engine returns your grade card within 60 seconds.

**What to submit:**
1. The name of the Project you used and a one-sentence description of what standing knowledge it contains (system prompt summary or document list — no need to paste the full system prompt)
2. A one-sentence description of the task you chose and why it requires the Project context
3. Your first prompt — exact text, as sent to Claude
4. Claude's full response to the first prompt — paste in full, do not summarise
5. Your second prompt — exact text, showing that it builds directly on Claude's first response (reference specific output from the first turn)
6. Claude's full response to the second prompt — paste in full, do not summarise
7. An 80-word reflection: where did the Project context add value specifically? Name the moment — a word, a phrase, a decision, a piece of knowledge — where you could see the Project doing work that a standard conversation wouldn't have done

**Where to submit:** Paste all seven items into the **Submission box** at the bottom of this lesson on the Claude Mastery platform. The AI grading engine returns your grade card within 60 seconds.

**Length guide:** Project description: one sentence. Task description: one sentence. Prompts: exact text, no minimum. Claude responses: paste in full. Reflection: exactly 80 words — not 60, not 120. The constraint is intentional.

**Pass mark:** 70 / 100. If you score below 70, the platform shows your grade card with feedback per criterion and activates a **Revise & Resubmit** button after a 24-hour cooldown.

**Rubric (100 points total — pass at 70):**
- **Specificity (25 pts)** — Is the task genuinely real — something the student actually needs done, not a demo prompt like "write something using my business context"? Does the task clearly require the Project's standing knowledge to work well — would Claude's output be meaningfully different without it?
- **Structure (25 pts)** — Are the two prompts genuinely sequential? Does the second prompt build directly on the first — continuing the same piece of work, not starting something new? Are all seven items present, clearly labelled, and in order?
- **Constraint clarity (25 pts)** — Does the second prompt reference specific output from the first? For example: a phrase Claude used that the student wants to adjust, a list Claude generated that the student wants to extend, a decision from the first turn that the second turn builds on. A second prompt that could have been sent without reading the first response signals the student is using Claude one-shot, not conversationally.
- **Outcome focus (25 pts)** — Does the reflection identify a specific moment where the Project context added value — not just "it knew my business" but a named piece of knowledge that changed the output? Does the student show they can see the difference between what the Project contributed and what a good prompt alone would have contributed?

## Notes for the AI grading engine

(Not shown to student)

- **Reward (high marks):**
  - The task is clearly real: a named client, a named document, a named situation — not "I asked Claude to write an email for my business"
  - The Project description shows the student has extended or set up something new, not just described their existing Project unchanged
  - The second prompt references specific output from the first — a sentence Claude wrote, a list item, a decision or phrasing from the first turn — proving the student read the first response before continuing
  - The reflection names a specific piece of Project knowledge (a term from the system prompt, a fact from an uploaded document, a communication standard) and traces how it appeared in the output — not generic claims like "it understood my business"
  - Excellent reflections often include a contrast: "A standard Claude conversation would have needed me to explain X — the Project already knew it, so the first response was immediately usable without that setup"
  - The two turns form a logical unit: turn one produces something, turn two does something further with it — a draft and then a revision, a list and then a prioritisation, a summary and then a follow-up action

- **Penalise (dock points):**
  - The task is a demo or test — "I asked Claude to introduce itself using my Project context" or "I tested whether Claude knew my rates"
  - The second prompt could have been sent without reading the first — it's a new task in the same Project, not a continuation of the first
  - Claude's responses are summarised or cut — the student must paste in full
  - The reflection is generic: "it helped a lot," "Claude understood my business," "I could see the context working" — no named specific
  - The Project description says "same as before" or is absent — no evidence the student extended their Project or created a second one
  - The task clearly doesn't need Project context — Claude would do it equally well in a standard conversation

- **Common 60–69 patterns:** The task is real and the two prompts are present, but the second prompt doesn't clearly build on the first — it's a related task, not a continuation. Or the reflection describes what happened without naming a specific moment where the Project knowledge appeared. In feedback: quote the second prompt and ask — what specific thing from Claude's first response does this reference? If the answer is nothing, that's the gap. Then quote the reflection and ask — which word or phrase in Claude's output came from the Project, not from your prompt? Show what a specific answer looks like.

- **Common 80+ patterns:** The task is a real piece of ongoing business work. The second prompt quotes or directly references something from Claude's first response — a list number, a phrasing decision, a recommended approach — and asks Claude to build on it or adjust it. The reflection names a specific term or instruction from the system prompt (or a fact from an uploaded document) and traces exactly where it appeared in the output. The student can articulate the difference between what the Project contributed and what their prompt contributed. Often the best submissions include a note like "I've added [X] to my Project document because this task showed me it was missing."

- **Feedback tone:** Direct, specific, kind. Quote the student's words. If the second prompt doesn't reference the first response, quote both and show exactly what a connecting reference would look like. If the reflection is generic, quote it and model what a specific version would sound like with one concrete substitution. End with one concrete next step: either a specific fix for resubmission, or a suggestion for the next Project task they should try based on what this submission reveals about their workflow.

- **Resubmission gating:** Check specifically that the second prompt has been revised to explicitly reference output from the first turn — this is the most common gap in first submissions. Check that the reflection now names a specific piece of Project knowledge rather than a general observation. If the student has also extended their Project in response to feedback, award credit for that explicitly and note what the extension adds.

## Industry personalisation hooks

The most natural task types for this deliverable vary by industry. Trades and construction: a quote in the first turn, followed by a revision that adjusts scope or terms based on the initial draft. Retail and e-commerce: a product description batch in the first turn, followed by a second turn that adapts the strongest option for a different channel. Financial services and bookkeeping: a client summary in the first turn, followed by a follow-up email that references specific figures or decisions from the summary. Health practices: a patient communication template in the first turn, followed by a version adjusted for a specific condition or age group named in the first response. Design, marketing, and agencies: a brief summary or creative direction in the first turn, followed by the first draft of copy or concepts that builds on it. Property management: a lease clause or inspection report in the first turn, followed by a tenant-facing summary of the key points. The deliverable brief is intentionally open — students choose from their own work — but industry-personalised examples on the platform page for this lesson can illustrate the two-turn structure for each of these common patterns.
