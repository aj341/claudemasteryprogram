## Lesson script

### More capability is not always better

Here is a belief that trips up a lot of people who've worked through this course: if Claude can do something more powerful, that's always the better option. Upload a file instead of pasting text. Build a Project instead of a standalone chat. Set up an automation instead of writing a prompt manually. More advanced means more effective, right?

Not always. Every extended capability comes with a cost that doesn't appear on the Claude pricing page. That cost is complexity — the setup time, the things that can now break, the mental overhead of maintaining something, and the context requirements that need to be right before the feature even works as intended.

A Project with a detailed system prompt and five uploaded reference documents is more capable than a basic chat. It's also more things to maintain, more failure points, and more time to set up correctly. When that Project is serving a recurring workflow you run three times a week — great investment. When it's for a task you do once a month — you've spent two hours of setup time on something a well-written prompt in a standard chat could have handled in three minutes.

The question isn't "what's the most powerful tool available?" The question is "what's the right tool for this specific task?" Those are different questions, and confusing them wastes time and introduces friction you don't need.

### The four-tier framework

Think of Claude's capabilities in four tiers, each with a clear use case.

**Tier 1 — Basic chat.** A single conversation, no setup, no uploads, no persistent memory. You open a chat, type a prompt, get a response, close it. This is the right tier for one-off tasks, quick questions, first drafts of things you'll refine yourself, and anything where the context fits in a single prompt. Most business owners underestimate how much they can get done at this tier with a well-crafted prompt.

**Tier 2 — Project + knowledge base.** A named Project with a system prompt and one or more uploaded reference documents. The right tier for recurring workflows where you always need the same context — your tone of voice, your service list, your client's details, your internal processes. The system prompt carries the standing instructions so you don't have to restate them every conversation.

**Tier 3 — File and vision processing.** Uploading documents, images, or screenshots into a conversation so Claude can read, analyse, or extract from them. The right tier when the task requires Claude to engage with content that doesn't fit neatly into a typed prompt — a competitor's PDF proposal, a supplier invoice, a photo of a whiteboard from a planning session.

**Tier 4 — API and automation.** Claude triggered by events rather than by you. A form submission arrives, an automation sends it to Claude, Claude drafts a response, the response goes somewhere — without you sitting there. The right tier for tasks you've already done manually many times, know work, and want to stop repeating.

Each tier up adds setup time and complexity. Each tier up is only worth that cost if the task genuinely needs what the higher tier provides. Most tasks — probably more than you think — belong in Tier 1 or Tier 2.

### The 3-question test

Before you reach for an extended capability, run this test. Three questions. If you can't answer yes to all three, stay at the simpler tier.

**Question 1: Does it save more than 30 minutes a week?**

Calculate the real number. If uploading and processing a document saves you ten minutes and you do it twice a week, that's twenty minutes. Not above the threshold — at least not enough to justify significant setup time. If you're responding to fifteen enquiry forms a week and a good automation saves you twelve minutes per response, that's three hours. Well above the threshold. The time saving has to be real and recurring, not theoretical and occasional.

**Question 2: Would a basic prompt achieve 80% of this?**

This is the hardest question to answer honestly. It requires you to actually try the simpler approach before concluding you need the more complex one. Many business owners skip straight to building a Project or setting up an automation because they assume a basic prompt won't be good enough. Often a properly structured Tier 1 prompt — with a role, a clear task, a format, and a length constraint — gets you to 80% of what the complex setup produces. If 80% is enough for this task, the complex setup is overhead you don't need.

**Question 3: Will you use this more than once a month?**

Frequency determines whether setup cost is worth paying. A Project you open three times a week amortises its setup cost quickly. A Project you open twice a year does not — and you'll spend more time remembering how it's configured than you'll save by having it. If the answer to this question is "less than monthly," that's a signal to use a well-written reusable prompt template instead, not a persistent setup with moving parts.

If the task clears all three — saves more than 30 minutes a week, a basic prompt won't get you to 80%, and you'll use it more than monthly — the extended capability is worth building. If it fails any one of them, stay simpler.

### Common traps: where over-engineering happens

Most over-engineering in Claude setups follows a few repeating patterns. Recognising them saves you time and frustration.

**Using vision when text would do.** Someone takes a screenshot of a document, uploads it as an image, and asks Claude to read and summarise it. This works — but Claude is reading pixels and reconstructing text. If the document exists as a copyable text file, copying and pasting the relevant section is faster and produces a more accurate result. Vision is valuable when the document genuinely only exists as an image. It's overhead when it doesn't.

**Common mistake → Better approach**

A business owner receives a supplier quote as a PDF. She screenshots the key pricing table and uploads the image so Claude can extract the figures. Claude reads the image, but misreads two numbers — "8" looks like "B" at that resolution.

The better approach: open the PDF, select and copy the pricing table text, paste it directly into the chat. If the PDF is scanned and not selectable, the image upload is the right call. If the text is selectable, use it. The rule: prefer machine-readable text over images when you have the choice.

**Uploading a PDF when pasting the key section is faster.** This is a subtle version of the same trap. You have a ten-page supplier contract, and you want Claude to review the payment terms clause. You upload the whole PDF. Claude processes ten pages to find the two paragraphs you need. The faster and more precise approach: open the contract, navigate to the payment terms section, copy those three paragraphs, paste them into the chat with your question. You get a sharper answer because Claude isn't parsing irrelevant content. You get it faster because you're not waiting on a file upload.

The principle: give Claude exactly what it needs for the task, not everything that's tangentially related.

**Building an automation when a template prompt is good enough.** Automations are powerful, but they're also the highest-complexity tier. They require setup, testing, maintenance, and a no-code tool subscription. Before you invest that, ask: could a well-designed prompt template — one you paste into a Tier 1 chat and fill in manually each time — produce the same output in two minutes instead of thirty? For a task you do twice a week, that's four minutes spent versus a weekend spent building and debugging an automation. If the manual version with a template prompt takes less than five minutes per run, the automation case is weak unless the volume is very high.

**Treating a Project as the default for everything.** Projects are useful. But they're not the right container for every task. If you're using a Project for a one-off task — writing a single press release, drafting a one-time proposal for an unusual client — the Project structure is overhead you didn't need. Open a standard chat, write a thorough prompt, get the output, close it. Save Projects for recurring workflows with consistent context requirements.

### Auditing your own Claude setup

The final skill this module builds is the ability to look at your actual Claude usage — not hypothetical use cases — and ask honestly: am I using the right tool for each job?

Pull up Claude and look at your recent conversations and any Projects you've built. For each recurring thing you use Claude for, ask yourself:

- What tier am I using?
- Does this task actually need that tier, or am I over-engineering out of habit?
- If I'm under-engineering — if I'm doing the same contextual setup manually every time in a basic chat — would a Project or a template prompt save me meaningful time?

The audit goes both ways. Over-engineering (using a complex setup for a simple task) wastes setup time and adds maintenance burden. Under-engineering (typing the same background context into every chat because you haven't set up a Project) wastes repetitive effort you've already done.

The goal is a setup where the tier matches the task. Simple tasks in basic chat. Recurring context-heavy tasks in Projects. Document-heavy tasks using file or vision input. High-volume repetitive tasks in automations — once they're truly ready for it.

You won't get this perfectly right on the first pass. Most people find at least one thing in their audit that belongs at a different tier. That's not a failure — that's the point of auditing. The value is in the correction.

A useful way to do the audit: for each task you use Claude for regularly, write down the tier you're using and the tier that seems like the right fit after running the 3-question test. If they match, you're calibrated. If they don't, you've found a change worth making.

This lesson's deliverable asks you to do exactly that — three tasks, real and current, each evaluated against the framework. One of those evaluations becomes a change you make now, not eventually.

## Worked examples

### Example 1: Basic chat is the right call

**Scenario:** A bookkeeper needs to draft a polite but firm payment reminder email for an overdue invoice. She does this a few times a month for different clients.

**Prompt:**
```
You are a professional bookkeeper writing a payment reminder email on behalf of a small business client. The email should be polite, clear, and firm — not apologetic, not aggressive. Include: the invoice number and amount, the original due date, a direct request for payment by a specific date (7 days from today), and payment method options (bank transfer or credit card via the link below).

Invoice number: INV-1084
Amount outstanding: $3,400
Original due date: 1 April 2026
Today's date: 21 April 2026

Payment link: https://pay.example.com.au/1084

Keep it under 120 words. No subject line needed.
```

**Why it works:** This task happens a few times a month, but it's quick to run manually with a template prompt like this one. The specifics change each time (invoice number, amount, date, link) — but the prompt structure is identical. No Project needed, no automation needed. Copy the template, fill in the variables, run it. Done in ninety seconds. The 3-question test: saves around five minutes per email — below the 30-minute-per-week threshold. A basic prompt gets to 100% of the required output. Less than monthly frequency per specific client. This belongs at Tier 1.

### Example 2: A Project is the right call

**Scenario:** A marketing consultant writes social media captions for three recurring clients — a physiotherapy practice, a café, and a landscaping business. Each client has a different tone and a different list of content topics they've approved.

**Prompt (inside a Project with client context in the system prompt):**
```
Write five Instagram caption options for the physio practice this week. The theme is winter wellness — specifically how staying active in cold weather helps prevent joint stiffness. Captions should be informative but warm, 80–100 words each, and end with a soft call to action (book online or call the clinic). No medical claims, no emoji overuse — maximum two per caption.
```

**Why it works:** The consultant runs this task every week for each client. The client tone, content rules, and approved topics live in the Project's knowledge base. Without the Project, she'd paste the same context — two or three paragraphs of client background — into every chat. With the Project, the standing context is always there. The 3-question test: saves her 45 minutes a week across three clients (retyping context). A basic prompt without the knowledge base would get her to maybe 60% — the tone calibration and content restrictions matter. She uses each Project at least weekly. All three questions pass. Tier 2 is right.

### Example 3: File processing is the right call — but only for the right section

**Scenario:** A small retail buyer wants to compare two supplier pricing proposals received as PDFs.

**Prompt:**
```
I'm comparing two supplier proposals for the same product line. Below are the pricing and terms sections from each — copied directly from the PDFs.

Supplier A:
Unit price: $4.20 ex-GST
Minimum order: 500 units
Payment terms: 30 days EOM
Freight: Included on orders over $2,000

Supplier B:
Unit price: $3.95 ex-GST
Minimum order: 750 units
Payment terms: 14 days from invoice
Freight: $180 flat rate per order under $3,000, free above

My average order value is around $2,500 and I reorder every six weeks.

Compare these two proposals for my situation. Tell me which is the better deal and why — show the working.
```

**Why it works:** The buyer didn't upload the full PDFs — she copied and pasted the two relevant sections. Claude gets exactly the information needed, nothing more. The prompt includes her specific context (average order value, reorder frequency), which makes the comparison actionable rather than generic. Uploading both full PDFs and asking "which is better?" would produce a vaguer answer, because Claude would be parsing terms and conditions that aren't relevant to the decision. Precision of input produces precision of output.

### Example 4: Automation is NOT the right call — yet

**Scenario:** A graphic designer wants to automate responses to new project enquiries sent through her website contact form. She launched her freelance business four months ago and has handled around twelve projects.

**Prompt she drafted for the automation:**
```
Read the enquiry below and write a response email that sounds like me, proposes a timeline, and gives a rough price range.
```

**Why automation isn't the right call:** The prompt has no role definition, no output format, no tone guidance, no pricing structure, and no constraint on length. After twelve projects, the designer doesn't yet have a stable sense of what a good response looks like — and she can't write it down clearly enough to configure Claude reliably. If this automation runs, it will produce inconsistent outputs that may misrepresent her pricing or promise timelines she can't meet. The 3-question test: saves maybe 10 minutes per enquiry — she gets three to four a month, so under two hours saved. A well-written manual prompt would get her to 90% of the output. Frequency is low (under ten a month). This task belongs at Tier 1 with a well-crafted template prompt — and back to Tier 4 consideration at the twelve-month mark, when her process is stable.

### Example 5: Recognising under-engineering

**Scenario:** An accountant uses Claude every day for client correspondence. Every time, she starts a new basic chat and types the same three-paragraph background: her firm's name, her communication style, the types of clients she serves, and her policy on recommending actions versus providing advice.

**What she does now:**
```
I'm an accountant at a small firm in Melbourne. We work with small business clients — mainly trades, hospitality, and retail. My communication style is direct and plain — no jargon, no hedging. I give practical guidance but always tell clients to confirm major decisions with their adviser...

[then the actual task]
```

**What she should do instead:** Set up a Project with that background in the system prompt — once. Every daily correspondence task then opens inside that Project with the standing context already loaded. No retyping. The 3-question test: she uses Claude for correspondence daily, so the recurring context restatement costs her ten minutes a day — more than 30 minutes a week. A Project gets her to 100% of what she currently produces, without the repetition. She uses it daily. All three questions pass. Tier 2 is the right call — and this is a case of under-engineering, not over-engineering. The fix is to move up a tier, not down.

## Graded deliverable

**Title:** Audit three tasks — pick the right tier for each

**Brief:** This deliverable is an audit of your actual Claude usage, not a hypothetical exercise. You will identify three real tasks you currently use Claude for — or have recently used Claude for — and evaluate whether you're using the right capability tier for each one. One of those three evaluations must result in a change you make now, not eventually. You'll show the change working: the updated approach and Claude's response.

This audit will likely surface at least one case of over-engineering and at least one case of under-engineering. Both are common, and identifying either one is a pass-worthy finding — the rubric rewards honest specificity, not "I learned to simplify."

Submit everything into the **Submission box** at the bottom of this lesson on the Claude Mastery platform. The AI grading engine returns your grade card within 60 seconds.

**What to submit:**

1. **Three task audit entries.** For each task:
   - **The task:** What you use Claude for — specific enough that someone could picture it (not "email writing" — "writing follow-up emails to clients after a project milestone sign-off")
   - **Current tier:** Which capability tier you use today (Tier 1 basic chat / Tier 2 Project + knowledge base / Tier 3 file or vision / Tier 4 API automation) — and a one-sentence description of how you use it
   - **Recommended tier:** Which tier you should use — with a specific justification based on the 3-question test (state each answer: saves more than 30 min/week? basic prompt gets 80%? used more than monthly?)
   - **What you'll change:** One concrete change — either moving to a higher tier (e.g., "set up a Project with my tone guide") or moving to a simpler tier (e.g., "stop uploading the full contract — paste only the relevant clause")

2. **One change demonstrated.** Pick the most impactful of the three changes. Do it now. Show the updated approach — the new prompt, the new setup, or the new method — followed by Claude's actual response. Don't describe what you would do. Show what you did.

3. **An 80-word reflection.** What did this audit reveal about your patterns? Identify the specific tendency — over-engineering, under-engineering, using vision when text would do, retyping context instead of using a Project — and name why you think it developed. Don't write "I learned to keep things simple." Write what you actually found.

**Where to submit:** Paste all three items into the **Submission box** at the bottom of this lesson on the Claude Mastery platform. The AI grading engine returns your grade card within 60 seconds.

**Length guide:** Each task entry: four labelled fields, as specific as needed. Demonstrated change: prompt in full (no bracket placeholders) + Claude's response pasted in. Reflection: exactly 80 words — the constraint is intentional.

**Pass mark:** 70 / 100. If you score below 70, the platform shows your grade card with feedback per criterion and activates a **Revise & Resubmit** button after a 24-hour cooldown.

**Rubric (100 points total — pass at 70):**

- **Specificity (25 pts)** — Are the three tasks real and current — not hypothetical? Is the justification for each tier decision grounded in the 3-question test with specific answers (not "it saves time" but "it saves 45 minutes a week because I restate client context in every chat")? Does the task description give enough detail to picture what the work actually involves?
- **Structure (25 pts)** — Does each of the three entries cover all four components: the task, the current tier, the recommended tier (with 3-question test answers), and the change? Is the demonstrated change clearly separated from the audit entries? Are all three submission items present?
- **Constraint clarity (25 pts)** — Is the demonstrated change actually shown — prompt in full, Claude's response pasted in — rather than described? Does the demonstrated prompt reflect the tier change the student identified (e.g., a Project system prompt if they moved to Tier 2, or a more targeted paste if they moved from file upload to direct paste)? Are bracket placeholders absent — was the prompt actually run?
- **Outcome focus (25 pts)** — Does the 80-word reflection identify a specific pattern — not just "I over-engineered" but which tasks, which tendency, and why? Does the student show they understand the difference between a finding of over-engineering and a finding of under-engineering? Is the reflection honest — does it match what the audit entries actually revealed?

## Notes for the AI grading engine

(Not shown to student)

- **Reward (high marks):**
  - All three tasks are named with enough specificity to picture the work — "writing Instagram captions for three recurring clients" not "social media content"
  - The 3-question test is answered explicitly for each task: each of the three questions (30 min/week threshold, 80% from a basic prompt, used more than monthly) is stated with a specific answer, not just a conclusion
  - The recommended tier differs from the current tier for at least two of the three tasks — all three identical is suspicious; a mix of "stay here," "move up," and "move down" is realistic
  - The demonstrated change shows a genuinely new approach — a Project system prompt written out in full, or a prompt that pastes a specific section rather than uploading a whole file — not just a minor edit to what they already did

- **Penalise (dock points):**
  - Any of the three tasks is described at the level of a category, not a specific task ("email writing," "research," "content creation" with no further detail)
  - The 3-question test is mentioned but not answered per task — a general reference to the framework without task-specific application
  - The "recommended tier" is the same as the "current tier" for all three tasks with justifications that don't engage with the test criteria — likely the student is rationalising, not auditing
  - The demonstrated change uses bracket placeholders or is described in future tense ("I would do this...") — it was not actually done
  - Claude's response is missing or summarised rather than pasted in full
  - The 80-word reflection is a generic summary of the lesson content ("I learned that simpler is better") without a specific personal finding
  - The reflection word count is substantially off — under 60 or over 110 words suggests the student didn't treat the constraint seriously

- **Common 60–69 patterns:** The three tasks are real but described loosely — enough to understand the category, not enough to picture the specific work. The 3-question test answers are implied ("this saves a lot of time") rather than stated ("this saves 45 minutes a week"). The demonstrated change is shown, but the prompt is close to what the student already used — the tier shift is claimed but not visible in the work. The 80-word reflection identifies a pattern but attributes it to "habit" without naming what the habit is or why it formed. In feedback: quote the task description and ask for one more specific detail — what kind of email, for which client type, how often. Ask the student to state each 3-question answer as a number or a yes/no. Quote the demonstrated prompt and ask: what changed from your previous approach?

- **Common 80+ patterns:** All three tasks are named at the level of a specific recurring workflow. The 3-question test answers are numerical or clearly stated: "saves 40 minutes a week," "a basic prompt gets me to about 60% — not enough because the tone calibration matters," "I do this daily." The tier recommendation is different from the current tier for at least two tasks, and the reasoning is grounded in the test rather than intuition. The demonstrated change is visibly different — if they're moving to Tier 2, there's a Project system prompt written in full; if they're moving down from file upload to paste, the prompt shows a specific pasted section. The 80-word reflection names a specific pattern, locates it ("I always reach for Projects because they feel more organised, even for one-off tasks"), and acknowledges what the audit found — often something surprising or uncomfortable. - **Feedback tone:** Direct, specific, and kind. Quote the student's task descriptions when showing what more specificity looks like. If the 3-question test wasn't answered per task, list the three questions and ask the student to apply them explicitly to each task in their resubmission. If the demonstrated change looks like the same approach with minor edits, name what you see and ask: what tier did you move to, and where is that visible in the prompt? End with one concrete next step — either a specific edit for resubmission, or (if they passed) the one audit finding worth acting on next.

- **Resubmission gating:** Check whether the task descriptions have become more specific — not just longer. Check whether the 3-question test is now answered explicitly per task. Check whether the demonstrated change shows a genuine tier shift — not just a rephrased version of the original. If the student changed which task they demonstrated, award credit if the new demonstration is more clearly grounded in the tier change they identified. Consistent improvement across all three criteria (more specific, more structured, more honest reflection) warrants a pass on resubmission even if the individual components aren't perfect.

## Industry personalisation hooks

The four-tier framework and the 3-question test apply across every industry represented in the AU/NZ small business cohort, but the most common failure modes vary by sector. Trades and construction businesses tend to under-engineer — they use Claude ad hoc in basic chat and retype the same project context every time, when a Project with their standard scope of work and communication style would save significant time. Professional services — accountants, bookkeepers, consultants — tend to over-engineer in a different direction: they build elaborate Projects for tasks they do infrequently, then forget the setup and end up doing the task in basic chat anyway. Retail and hospitality operators often default to the vision tier — screenshotting price lists, menus, and supplier sheets — when the underlying text is selectable and paste would be faster and more accurate. Marketing consultants and designers are most likely to default to automation thinking before their processes are stable enough to automate reliably. Tailor feedback and examples to the student's stated industry where it appears in their submission.
