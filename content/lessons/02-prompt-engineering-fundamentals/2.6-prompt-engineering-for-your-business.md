Each lesson was a layer. This lesson is where you see the whole structure at once.

This is not another technique to learn. It is a synthesis lesson. The purpose is to take the five layers you've built and assemble them into something that works for your actual business, every week, without starting from scratch.

### The Prompt Library concept

The most practical thing you can build from Module 2 is a Prompt Library. Not an archive of every prompt you've ever sent — a curated set of tested, working prompts for the tasks you run on a recurring basis.

Here is what that means in practice.

Think about the three to five tasks you do most frequently that involve writing, drafting, summarising, or communicating. Client emails. Meeting summaries. Quote follow-ups. Social media posts. Internal updates. Policy documents. Supplier proposals. Whatever lands on your desk every week.

For each of those tasks, you now have the skills to write a prompt that works. You've tested prompts in this module. Some produced good outputs. Some needed revision. After that work, you know what the prompt for that task looks like when it's right.

A Prompt Library is where you keep those working prompts — stored, named, ready to deploy.

Think of it as a starter kit. You don't write the prompt from scratch next Monday. You open the library, find the matching template, fill in the variable parts, and send it. The hard thinking — what context to include, what format to specify, what constraints to set, what to exclude — is already done. You did that thinking once. Now you use the result.

This is how Claude shifts from something you occasionally try to something that consistently saves you time. Without the library, you're rebuilding from memory every session. With it, you're running a tested system.

The library doesn't need to be elaborate. A Google Doc with three sections. A note in Notion. A simple table in Word. Wherever you'll actually go back to it. The format is irrelevant. What matters is that the prompts are written out in full, tested against real tasks, and annotated with when and how to use them.

### How to turn any working prompt into a durable template

Not every good prompt needs to become a template. One-off tasks don't need templating — you write the prompt, use it, move on.

But when you find a prompt that works well for a task you run more than twice a month, it's worth the three minutes to extract a template from it. Here's how.

**Step 1: Write out the full working prompt as-is.** Don't edit it yet. Copy it exactly as it was when it produced a good output. This is your proof of concept. If you start editing before you capture it, you risk removing the thing that made it work.

**Step 2: Identify the variable parts.** Read through the prompt and mark every piece of information that will change between uses. The client's name. The specific task or project. The date or timeframe. The product or service in question. Replace each of those with a short, clear label in brackets:

```
[client name], [project type], [specific ask], [tone: formal/conversational], [word count]
```

Everything that stays constant — your business context, the format instruction, the constraints, the structure — stays as written. The variable labels are the only things you change each time you use the template.

**Step 3: Add a usage note.** Under the template, write two or three sentences that describe when to use this prompt and what to watch for. Which version of the task is this template suited to? Are there situations where it won't work — where the task is too complex, or the audience is different enough that you'd need a separate template? A good usage note saves you the moment of hesitation next time you open the library and wonder whether this is the right one.

That's it. Three steps, three minutes, and you have a template you can use for the next two years without rebuilding it.

### Common mistake → better approach

**Mistake:** Saving a prompt that "sort of worked" and calling it a library entry.

A near-miss prompt looks like a working prompt. It produced an output that was mostly fine. You used it — with some editing. You saved it.

But next time you pull it from the library, you send it, and the output has the same gap it had last time: too formal, or too long, or missing a key piece of structure. You edit again. And again. The "library" is just an archive of rough drafts.

**Better approach:** Before you save any prompt as a library entry, apply one final test. Send it to Claude right now with a real but hypothetical version of your task. Read the output. Ask: would I send this to a client or colleague with only minor copy-editing, not a structural rewrite? If yes, save it. If no, fix the one thing that's still off — a missing constraint, a format instruction that's too vague, a context gap — and test again. Only save the version that passes the test.

Library entries should be production-ready. Not perfect — production-ready. There's a difference. Perfect means the output needs zero editing. Production-ready means the output needs the kind of light copy-edit any professional document gets before it goes out — not a structural rethink.

### From figuring it out to using it as a tool

There is a specific mindset shift that happens between Module 1 and the end of Module 2. It's worth naming explicitly, because if you don't notice it happening, you might not take it seriously.

At the start of Module 1, Claude was something you were testing. You were learning what it could do. You were building intuition about how to write prompts. That's appropriate for the start of a course.

By now, that phase is over. You're not figuring out prompts anymore. You've done the work. You've tested real prompts on real tasks. You've learned what makes a prompt work and what causes a revision round. You've built templates.

The mindset that comes after is simpler: Claude is a tool in your business, and your Prompt Library is how you run that tool. You use the library the same way you use any other professional tool — reliably, deliberately, without reinventing the process each time.

This doesn't mean prompt work is finished. Your library will grow. You'll add new templates as you take on new task types. You'll refine existing ones when a usage note reveals a gap. You'll occasionally retire a template that no longer fits your business.

But that's maintenance, not discovery. The discovery phase of Module 2 ends here. What comes next is use.

Here's the practical version of that mindset: when you sit down next Monday with a task to complete, don't open Claude.ai and start typing from memory. Open your Prompt Library first. Find the matching template. Fill in the variables. Send it. Read the output. If it's production-ready, use it. If it needs one revision, send the revision. That's the workflow.

You've built it. Now run it.

### Synthesis: the five principles as a system

Before you go to the deliverable, it's worth running the five Module 2 principles through a single lens to see how they connect.

**Components** (2.1): The base layer. Every prompt needs context, task, format, and constraints. Miss one and the output drifts. These four components are the non-negotiables — every library template you build should include all four.

**Context** (2.2): The depth layer. Context isn't just background — it's the frame through which Claude interprets everything else. The more precisely your context describes the specific situation (not just your general business), the less guesswork appears in the output. Good context is the single highest-leverage change most people can make to their prompts.

**Templates** (2.3): The efficiency layer. A tested template is the payoff for doing prompt work once. Without templates, every session starts from scratch. With them, you deploy tested structure and put your cognitive effort into the variable parts — the actual task — not the scaffolding around it.

**Advanced techniques** (2.4): The precision layer. Role assignment, negative constraints, multi-step framing — these are refinements you reach for when the standard structure isn't producing the precision you need. They're not defaults. They're tools for specific problems.

**Iteration** (2.5): The feedback layer. No prompt is final after one use. Iteration is how prompts improve over time. The point is not to iterate endlessly — it's to iterate efficiently, once or twice per session, and capture what you learn so the next version of the template is better.

Together, these five layers describe a complete prompting practice. Not a collection of tricks. A practice that compounds. Every good session makes your library better. Every refined template makes the next similar task faster. That's the structure you've built.

Now prove it.

## Worked examples

### Example 1: Prompt Library entry — client proposal for a bookkeeping firm

**Scenario:** A bookkeeper wants a reusable template for drafting the written proposal section of new client engagements.

**Prompt:**
```
I run a bookkeeping practice in Melbourne called Clearwater Accounts. We work
with small businesses in retail and hospitality — typically 2 to 15 staff,
annual turnover between $500k and $5m. Our clients need BAS preparation,
monthly reconciliation, and payroll processing. We don't do tax returns or
financial planning — we refer those out.

Our tone with clients: plain English, professional, never condescending.
We're the practice that explains things without making people feel stupid.

Draft the written proposal section for a new client engagement with
[client name], a [business type] in [suburb]. The scope covers [list of
services]. Estimated monthly fee: $[amount].

Format: three short paragraphs. Para 1: what we'll do for them and why it
matters. Para 2: how we work (monthly rhythm, what they need to provide, how
we communicate). Para 3: next steps and a clear call to action. Professional
but warm. Under 300 words. Do not include pricing tables or service tier
descriptions — this is the written narrative only.
```

**Usage note:** Use for any new client proposal where you're providing the written narrative. Not suitable for proposals that require a detailed scope change or where the client relationship is already established — write those from scratch. Fill in the four variables in brackets before sending.

**Why it works:** The template carries full business context, a clear three-paragraph structure, a word count, an explicit exclusion ("not pricing tables"), and usage guidance. The variable brackets are clear and minimal — only the genuinely changing information is left open.

### Example 2: Prompt Library entry — quote follow-up for a landscaping business

**Scenario:** A landscaping company owner wants a template for following up on residential quotes after 5–7 business days of silence.

**Prompt:**
```
I run a residential landscaping business on the Sunshine Coast. We do garden
design and installation for homeowners — jobs typically range from $8,000 to
$40,000. Our work is considered and detailed; we're not the cheapest, and
we don't try to be.

Client communication style: direct, friendly, no pressure. We let the work
speak for itself. We never use phrases like "don't miss out" or "limited
availability."

Write a follow-up email for a quote sent to [client name] [number of days]
ago for [brief job description], quoted at approximately $[amount]. The email
should: check in without pressure, offer to answer questions or walk through
any part of the quote, and leave the decision entirely with the client.
Under 120 words. Sign off with [owner first name].
```

**Usage note:** Use for first follow-up only, 5–10 business days after sending the quote. If the client still hasn't responded after a second follow-up, do not use this template — phone instead.

**Why it works:** The explicit "do not say" constraint ("don't miss out," "limited availability") protects the business's voice without requiring multiple revision rounds. The usage note prevents the template from being misapplied — it's for a first follow-up only, and the note makes that clear.

### Example 3: Prompt Library entry — internal weekly update for a physio clinic

**Scenario:** A physiotherapy clinic principal wants a template for her Friday afternoon all-staff summary email.

**Prompt:**
```
I'm the principal of a physiotherapy clinic in Canberra — 3 physios, 1 exercise
physiologist, 2 reception staff. We see private, WorkCover, and NDIS clients.

This is an internal communication — it goes to my staff only, not clients.
Tone: direct, practical, collegial. No corporate language. I write to my team
the way I'd talk to them in the kitchen.

Draft the weekly all-staff update email for Friday [date]. Cover the following:
[paste bullet points from notes: wins this week, any operational changes,
reminders for next week, one forward-looking item].

Format: short intro line, then each topic as a brief paragraph — 3 to 5
sentences maximum per topic. Plain subject line that tells them what's inside,
not a clever headline. Total length: under 250 words.
```

**Usage note:** Works best when you provide at least 3 bullet points in the variable section — sparse notes produce a thin email. If you have more than 5 topics, prioritise before pasting: this template is for a weekly snapshot, not a detailed brief.

**Why it works:** The role context (principal, internal comms) and tone description ("the way I'd talk to them in the kitchen") are specific enough that Claude calibrates immediately. The format instruction is detailed — per-paragraph length, subject line guidance, total word count — which prevents the template from producing a wall of text from a few loose bullet points.

### Example 4: Prompt Library entry — product description for an online homewares retailer

**Scenario:** An online homewares store owner wants a reusable prompt for writing product descriptions for new stock arrivals.

**Prompt:**
```
I run an online homewares store called Hearthside Home, based in Adelaide.
We sell considered, quality homewares — mostly Scandinavian and Japanese
influenced design. Mid-range to premium pricing. Our customers are design-
conscious adults aged 30–55 who are buying for their own homes, not gifts.

Tone: warm, specific, never breathless. We describe what things are actually
like, not how they'll make you feel in an aspirational sense. We don't use
words like "luxurious," "stunning," "perfect," or "must-have."

Write a product description for: [product name].
Key details: [material, dimensions, available colours or finishes, care
instructions if relevant].
Category: [e.g. cushions, ceramics, lighting].
Price point: $[amount].

Format: 2 short paragraphs. Para 1: what it is and what makes it worth
buying. Para 2: practical details (dimensions, materials, care). Under 100
words total. Written for the product page — not a social post.
```

**Usage note:** Supply all details in the variable section before sending. Do not use for sale or promotional items — the tone in this template is not suitable for promotional copy; write those separately.

**Why it works:** The explicit word-ban list ("luxurious," "stunning," "perfect," "must-have") is the key constraint that protects the brand voice. The two-paragraph structure, with specific purpose for each paragraph, prevents Claude from merging practical details and descriptive copy in a way that makes the description harder to scan.

### Example 5: Prompt Library entry — meeting summary for a business consultant

**Scenario:** A management consultant wants a template for summarising client discovery meetings for internal records and client sign-off.

**Prompt:**
```
I'm an independent management consultant based in Brisbane. I work with
businesses from 20 to 200 staff on operational efficiency and organisational
structure. Most of my clients are privately owned.

This is a meeting summary — it goes first to me as an internal record, then
to the client for sign-off. It needs to be readable by both audiences: precise
enough for my records, plain enough that the client's team can confirm accuracy
without needing to decode it.

Summarise the following meeting notes into a structured summary.

Meeting: [client name] — [meeting type, e.g. discovery session / project
review / scope discussion]
Date: [date]
Attendees: [list]
Notes: [paste raw notes below]

Format:
1. Meeting purpose (1 sentence)
2. Key discussion points (bullet list, 4–7 points)
3. Decisions made (bullet list — leave blank if none)
4. Action items (table: action, owner, due date)
5. Next meeting (date and agenda if confirmed)

Tone: precise and professional. Plain sentences. No filler phrases. Do not
infer or add information not present in the notes — if something is unclear,
flag it with [CHECK] rather than guessing.
```

**Usage note:** The [CHECK] instruction is important — keep it in the template. It prevents Claude from filling gaps with plausible-sounding but inaccurate content, which is especially important when the summary goes to the client for sign-off. Works for meetings up to 2 hours of notes; for longer sessions, split the notes into sections and summarise each separately.

**Why it works:** The [CHECK] flag instruction is an advanced constraint that prevents hallucination in a high-stakes context. The dual-audience note (internal record AND client sign-off) calibrates Claude's register without requiring two separate templates. The action item table format is specified explicitly — action, owner, due date — which forces structured output rather than a loose list.

**Where to submit:** Paste all three sections into the **Submission box** at the bottom of this lesson on the Claude Mastery platform. The AI grading engine returns your grade card within 60 seconds.

**Length guide:** Prompt Library table: as long as your three full prompts require — no shortcutting. Live outputs: paste in full, no minimum or maximum. Reflection: 150 words exactly or within 10 words either side.

**Pass mark:** 70 / 100. If you score below 70, the platform shows your grade card with feedback per criterion and activates a **Revise & Resubmit** button after a 24-hour cooldown.

**Rubric (100 points total — pass at 70):**
- **Specificity (25 pts)** — Are all 3 prompts fully written and grounded in the student's real business — not generic, not hypothetical, not interchangeable with any other business? Do the three prompts cover genuinely different task categories (e.g. not three versions of "write an email")? Prompts that could have been written by anyone in any industry score low regardless of structure.
- **Structure (25 pts)** — Does each library entry include the task category, the full prompt text, and usage notes? Is the full prompt text actually submitted — not a paraphrase or summary? Does each entry demonstrate that the student understands what a reusable template looks like versus a one-off prompt?
- **Constraint clarity (25 pts)** — Do all 3 prompts include format instructions, specific constraints, and context that demonstrate Module 2 skills? Look for: explicit word counts or length limits, tone instructions beyond vague adjectives, negative constraints or exclusions, format specifications (headings, tables, paragraph structure), and relevant business context that shapes the output. Prompts with only a task and a vague context score in the 10–15 range for this criterion.
- **Outcome focus (25 pts)** — Does the reflection show module-level synthesis — can the student articulate the core principles and how they connect, not just list techniques? A strong reflection explains how prompting works as a system, not just what the individual components are. It should be understandable to a novice: specific, clear, and grounded in what the student actually learned — not a generic summary of AI.

## Notes for the AI grading engine

(Not shown to student)

- **Reward (high marks):**
  - All three prompts are grounded in the student's specific business — industry, clients, voice, task context are all evident from reading the prompt alone
  - The three task categories are genuinely distinct — not three email drafts, not three social posts; the student has identified three different kinds of work
  - Every prompt demonstrates at least three Module 2 techniques: format instruction, at least one specific constraint, context beyond just business background (task-level or audience-level context)
  - Usage notes show judgment — the student knows when the template applies and when it doesn't, and says so explicitly
  - Live outputs are pasted in full — the grader can see that the prompt actually produced a useful, on-brief result
  - Reflection explains the system — why context matters, how constraints save revision time, why templates compound — not just "I learned about prompts"
  - Excellent reflections use language the student's peers would actually understand, grounded in real examples from their own prompts

- **Penalise (dock points):**
  - Any prompt that is generic — could apply to any business, with no named industry, audience, or business-specific constraint
  - Task categories that are effectively the same category (e.g. three client communication prompts with different audiences)
  - Prompts that are structural outlines or descriptions of a prompt rather than the full written prompt
  - Live outputs that are summarised, paraphrased, or absent — if the output isn't there, the prompt cannot be evaluated as production-ready
  - Reflection that lists Module 2 techniques without explaining how they connect or why they matter
  - Reflection under 130 words or over 175 words without clear justification

- **Common 60–69 patterns:**
  - The prompts are real and business-specific, but lack Module 2 techniques — no explicit format instructions, constraints are vague ("professional tone"), context is only business-level with no task-level specificity. The student has clearly used Claude, but hasn't applied what they learned. In feedback: quote one of their prompts back, show exactly where to add a format instruction and one specific constraint, and explain what that change would do to the output. Ask them to compare their submitted output to what they'd expect if those additions were in place.
  - Alternatively: prompts are technically strong but two of the three task categories are too similar — the student is showing depth in one area but not breadth across their business. In feedback: name the overlap directly and suggest a genuinely different task category from their business.

- **Common 80+ patterns:**
  - All three prompts read like they came from someone who has actually tested them — the output is specific, the constraints are calibrated, the usage notes show real operational judgment ("use for first follow-up only, not if client has already declined once")
  - The reflection is teaching material, not a course summary — the student explains prompt structure through the lens of their own experience, uses a concrete example, and makes an argument for why the system works, not just what it contains
  - Task categories span meaningfully different areas of the business — external communication, internal operations, and something process-related, for instance
  - Live outputs show that the prompts actually work at a production level — the grader can read them and see that they'd save time over writing from scratch

- **Feedback tone:** Direct, specific, kind. Quote the student's own prompt text when identifying what's working or what needs to change — never give generic feedback. If a constraint is missing, name exactly what kind of constraint and where in the prompt to add it. End with one concrete next step: either what to fix in the resubmission, or, for high scorers, a genuine observation about where their library will be most valuable in their day-to-day work.

- **Resubmission gating:** Compare the resubmission to the original. Check for three specific improvements: (1) did the prompts gain specificity — more precise constraints, more task-level context, or a new task category that fixes an overlap? (2) did the live outputs improve — is the new output more evidently production-ready than the first? (3) did the reflection deepen — does it now explain the system rather than list the parts? Name each improvement explicitly. If only one or two improved, acknowledge the progress and name the remaining gap with specific language. If the resubmission is the original with minor additions, say so and name what still needs to change.

## Industry personalisation hooks

The five worked examples cover: professional services bookkeeping (proposal narrative), trades/outdoor (landscaping quote follow-up), health practice (physiotherapy all-staff update), product retail (homewares product description), and consulting (management consultant meeting summary). In personalised variants, the examples closest to the student's own industry should appear first. The Prompt Library deliverable is inherently self-personalising — students supply prompts from their own business — so the instructional content is universal. The most useful personalisation lever is in the brief: the parenthetical task type suggestions for "three genuinely different task categories" can be swapped to examples from the student's industry (e.g. for a health practice: appointment reminders, clinical handover notes, and supplier communications; for a retail business: product copy, supplier emails, and social captions). The usage note concept in the three-step extract process is the highest-value teaching beat for non-developer audiences — it's the difference between a prompt archive and a working library.
